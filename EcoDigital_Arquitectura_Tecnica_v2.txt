--------------------------------------------------------------------------------
01  VISION GENERAL DEL SISTEMA
--------------------------------------------------------------------------------

EcoDigital (identificador de produccion: EcosSecial) es un ecosistema
administrativo nativo en la nube, aumentado con inteligencia artificial,
disenado para la gestion integral de consultorios de cirugia especializada.
Centraliza el manejo de expedientes de pacientes, historiales clinicos,
agendamiento, almacenamiento documental, control de acceso basado en roles
y flujos de trabajo asistidos por IA.


FASES DEL PROYECTO
------------------

  Fase 1 (Actual)
    Dashboard Administrativo Desktop: gestion de datos, RBAC,
    automatizacion IA.
    Estado: Activo

  Fase 2
    Aplicacion movil para equipo clinico; presencia web profesional.
    Estado: Planeado

  Fase 3
    App movil para pacientes; integracion AVI en WhatsApp.
    Estado: Planeado


RESTRICCIONES OPERATIVAS
-------------------------

  - Todos los datos de pacientes son informacion clinica sensible, sujeta
    a estandares de confidencialidad equivalentes a HIPAA.
  - Conexion a internet requerida: minimo 250 Mbps para rendimiento optimo
    en operaciones cloud.
  - El Desktop Shell opera bajo un modelo edge-hybrid; la inferencia IA se
    ejecuta localmente cuando no hay conectividad.
  - Sin dependencia de APIs propietarias de OpenAI. Cero costo por token
    en modo escritorio.
  - Todas las eliminaciones son borrado suave (Soft Delete); ninguna
    entidad se elimina permanentemente.


DOMINIOS FUNCIONALES PRINCIPALES
----------------------------------

  Gestion del Ciclo de Vida del Paciente
    Registro demografico, historial clinico, archivado.

  Agendamiento de Citas
    CRUD completo con deteccion de conflictos de horario.

  Gestion Documental
    Almacenamiento binario en nube con metadatos relacionales.

  Identidad y Control de Acceso
    RBAC multi-capa con auditoria inmutable WORM.

  Capa de Asistencia IA
    Generacion de texto, resumenes, borradores de correo, recordatorios.

  Desktop Shell
    Wrapper Electron con IA local, Sidebar y apps nativas integradas.

  Sidebar Interactivo
    Panel lateral con funciones fijas, apps nativas y WebViews custom.

02  FILOSOFIA DE ARQUITECTURA
--------------------------------------------------------------------------------

P1 — Unica Fuente de Verdad
  PostgreSQL (Cloud SQL) es el unico almacen persistente para todas las
  entidades estructuradas. No se permiten motores SQL secundarios. El
  estado de la IA, los registros de auditoria y el estado de la aplicacion
  persisten dentro de esta instancia.

P2 — Contrato API-First
  Todas las interfaces de usuario (Desktop Shell, Template Web, apps moviles
  futuras) se comunican exclusivamente a traves de la API REST versionada
  en /api/v1/*. Ninguna capa cliente puede establecer conexiones directas
  a la base de datos.

P3 — Infraestructura Scale-to-Zero
  Las instancias de Cloud Run mantienen min_instances: 0. El sistema no
  incurre en costo de computo sin trafico activo. Cloud SQL es el unico
  componente con costo fijo de uptime.

P4 — Defensa en Profundidad
  Perimetro de red: API Gateway + VPC Connector.
  Capa de aplicacion: validacion JWT + middleware RBAC.
  Capa de datos: cifrado a nivel de campo en columnas sensibles.
  Capa de auditoria: logging inmutable WORM.

P5 — Persona IA Unificada
  El usuario percibe un unico asistente IA sin importar el modelo que
  ejecute la inferencia. El enrutamiento entre Gemma 270M local, Gemma 3
  en nube, y modelos especializados (MedGemma, Gemini Nano) es invisible
  para la capa de interfaz.

P6 — Soft Delete Universal
  Ninguna entidad se elimina permanentemente del sistema. Todas las
  operaciones DELETE activan is_deleted=1 o status='Archivado'. Esto
  preserva la continuidad de auditoria y permite la recuperacion de
  expedientes clinicos.

3.2  GRUPOS DE FUNCIONES CUSTOM (INTEGRACIONES WEBVIEW — USUARIO)
------------------------------------------------------------------

Las integraciones custom son WebViews gestionadas completamente por el usuario.
El sistema provee una interfaz de gestion que permite AGREGAR, ELIMINAR y
REORDENAR aplicaciones web como si fueran aplicaciones instaladas.

Cada WebView opera en un proceso de navegador aislado con su propia sesion
y almacenamiento local.

Mecanismo tecnico:
  - Persistencia del registro de apps : LocalStorage + archivo JSON en userData de Electron
  - Aislamiento de sesion por WebView  : Electron partition: persist:app-{slug}
  - Proceso de renderizado             : BrowserView separado por app
  - Tabs en background                 : Frozen Tabs — proceso suspendido, memoria liberada
  - Seguridad de contexto              : contextIsolation:true, nodeIntegration:false
  - Maximo de apps instaladas          : Sin limite tecnico (recomendado <20)
  - Agregar nueva app                  : Usuario provee URL, nombre, icono
  - Verificacion de URL                : Validacion de dominio + HTTPS requerido

FLUJO DE TRAFICO — SOLICITUD AUTENTICADA
-----------------------------------------

  Accion del Usuario (UI Desktop)
    → IPC Electron (IA local) o solicitud HTTPS al API Gateway
    → Nginx (puerto 443, terminacion TLS 1.3)
    → API Gateway (enrutamiento OpenAPI spec, validacion IAM)
    → Cloud Run backend (solo ingress interno)
    → Frappe: middleware auth → verificacion RBAC → handler
    → PostgreSQL (Cloud SQL via IP privada VPC)
    → Respuesta (JSON) → renderizado Frontend

PIPELINE DE PROCESAMIENTO DE SOLICITUDES
-----------------------------------------

  1. Solicitud HTTPS entrante llega al API Gateway.
  2. API Gateway valida token; enruta al Cloud Run por ingress interno.
  3. Nginx (Cloud Run) hace reverse-proxy al worker de Frappe.
  4. Router Frappe encuentra el handler decorado con @frappe.whitelist().
  5. Middleware de auth decodifica JWT; resuelve identidad y rol del usuario.
  6. Middleware RBAC verifica Custom DocPerm para recurso y operacion.
  7. Handler ejecuta operaciones ORM de DocType contra PostgreSQL.
  8. Hooks on_update/on_trash disparan: escritura tabAudit Log + Cloud Logging.
  9. Respuesta JSON serializada y devuelta al cliente.

PATRON DE BORRADO SUAVE (SOFT DELETE)
--------------------------------------

  def soft_delete(doctype, name, user):
      doc = frappe.get_doc(doctype, name)
      doc.is_deleted = 1
      doc.status = "Archivado"
      doc.save(ignore_permissions=True)
      # on_update hook dispara automaticamente -> auditoria registrada

CONDICIONES DE ERROR COMUNES
------------------------------

  400 Bad Request            : Datos de entrada mal formados
  401 Unauthorized           : JWT invalido, expirado o ausente
  403 Forbidden              : Rol sin permiso para la operacion
  404 Not Found              : Recurso no encontrado o archivado
  409 Conflict               : Conflicto de horario en citas

  429 Too Many Requests      : Limite de tasa excedido
  500 Internal Server Error  : Error no manejado en backend

09  ARQUITECTURA DE SEGURIDAD
--------------------------------------------------------------------------------

SEGURIDAD DE TRANSPORTE
------------------------

  Todo el trafico cliente-nube transita por HTTPS forzado en el API Gateway
  con TLS 1.3 y certificados gestionados por Google. Puerto 443 es el unico
  puerto de acceso publico. No se aceptan conexiones HTTP en produccion.


FLUJO DE AUTENTICACION
-----------------------

  1. Cliente envia credenciales a POST /api/v1/auth/login.
  2. Backend valida contra hash bcrypt en tabUser.
  3. Backend emite JWT firmado (JWT_SECRET desde Secret Manager).
  4. Todas las solicitudes siguientes incluyen JWT en cabecera Authorization.
  5. Middleware Frappe valida firma y expiracion del JWT en cada solicitud.
  6. Renovacion disponible via POST /api/v1/auth/refresh.


CIFRADO DE DATOS
-----------------

  Datos en transito      : TLS 1.3 (todo el trafico externo via API Gateway)
  Datos en reposo SQL    : Cifrado server-side GCP con claves gestionadas
  Datos en reposo GCS    : Cifrado server-side GCP con claves gestionadas
  Campos sensibles       : Modulo encrypt Python de Frappe aplicado a
                           mobile_number, email, insurance_number
  Contrasenas            : Hash bcrypt; texto plano jamas almacenado
  Acceso a documentos    : URLs pre-firmadas TTL 15 min; rutas GCS nunca
                           expuestas al cliente


AUDITORIA — WORM DOBLE CAPA
-----------------------------

  Capa 1 — tabAudit Log (allow_edit=0)
    Hooks on_update y on_trash de Frappe escriben entradas estructuradas
    en cada mutacion de DocType. Ningun endpoint acepta UPDATE ni DELETE
    contra esta tabla.

  Capa 2 — GCP Cloud Logging (ecossecial-audit)
    Payloads JSON estructurados reenviados a Cloud Logging.
    No pueden modificarse ni eliminarse por ninguna interfaz o API.

  Esta arquitectura dual garantiza integridad de auditoria incluso
  ante una compromision a nivel de base de datos.


SEGURIDAD DEL DESKTOP SHELL

CASOS DE PRUEBA CRITICOS
-------------------------

  Autenticacion:
    - Login exitoso retorna JWT valido con claim de rol correcto.
    - Login fallido retorna HTTP 401; evento escrito en tabAudit Log.
    - JWT expirado rechazado con HTTP 401.

  Enforcement RBAC:
    - Rol Asistente no puede acceder a GET /api/v1/logs. Esperado: 403.
    - Rol Invitado no puede ejecutar POST /api/v1/patients. Esperado: 403.
    - Admin Principal accede exitosamente a todos los endpoints.

  Integridad Soft Delete:
    - DELETE /patients/{id} activa is_deleted=1; paciente ausente en lista.
    - Registro recuperable via GET /patients/{id} directo.
    - Entrada de auditoria creada con tipo_operacion = "eliminar".

  Ciclo de vida del documento:
    - Upload retorna signed_url valida con TTL < 15 minutos.
    - Archivo descargado coincide byte a byte con el subido.
    - Documento soft-deleted retorna HTTP 404 en recuperacion.

  Inmutabilidad de auditoria:
    - Ningun endpoint acepta PUT o DELETE contra tabAudit Log.
    - UPDATE directo en BD bloqueado por allow_edit=0 de Frappe.

13  GLOSARIO TECNICO
--------------------------------------------------------------------------------

API Gateway
  Punto de entrada GCP que enruta trafico HTTPS a Cloud Run y termina TLS.
  Perimetro de red publica del sistema. Aplica enforcement IAM en cada
  solicitud antes de reenviarla al backend.

Audit Log (WORM)
  DocType tabAudit Log con allow_edit=0. Registra cada mutacion del sistema
  con snapshots de estado anterior y posterior. Inmutable por diseno;
  replicado a GCP Cloud Logging.

bench
  CLI de orquestacion de Frappe. Gestiona sitios, apps, migraciones de schema,
  builds de assets y reinicios de procesos.

Black.Editor
  Editor PDF e IA asistida embebido en el Template Web. Soporta dictado,
  traduccion y resumen via inyeccion IPC de IA.

Black.View
  Modulo visor de imagenes medicas dentro del Template Web. Integracion
  MedGemma reservada para Fase 2+.

BrowserView
  API de Electron para renderizar contenido web en una vista subordinada
  aislada del BrowserWindow principal. Base tecnica de las WebViews del Sidebar.

Cloud Run
  Runtime de contenedores serverless de GCP. Aloja el backend. Escala a cero
  cuando esta inactivo; sin costo de computo sin trafico.

Cloud SQL
  Servicio de BD relacional gestionado de GCP. Aloja la instancia PostgreSQL 16.
  Unico almacen de datos estructurados del sistema.

Context Isolation
  Propiedad de Electron (contextIsolation:true). Previene que el renderer
  acceda a APIs de Node.js directamente. Hace de preload.js la unica
  superficie IPC.

DocType
  Unidad de abstraccion del modelo de datos de Frappe. Corresponde a una
  tabla de BD e incluye ORM, validacion, permisos y definiciones UI.

Frozen Tab
  Tab de Electron en background cuyo proceso renderizador esta suspendido
  para liberar memoria. Se reactiva al volver a ser seleccionado.

IPC (Inter-Process Communication)
  Mecanismo de Electron para comunicacion entre proceso principal y renderer.
  Expuesto via contextBridge en preload.js. Unico puente entre IA local y UI web.

JWT (JSON Web Token)
  Token de autenticacion sin estado emitido en login. Contiene identidad
  y rol del usuario codificados. Validado en cada solicitud API por middleware
  de Frappe.

llama.cpp
  Motor de inferencia en C++ para modelos LLM cuantizados. Backend de
  ejecucion para Gemma 270M en el Desktop Shell via binding nativo Node.js.

Nginx
  Proxy inverso frente a la aplicacion Frappe. Maneja terminacion SSL y
  enrutamiento interno en el entorno de servidor.

PITR (Point-in-Time Recovery)
  Capacidad de respaldo de Cloud SQL que permite restaurar la BD a cualquier
  punto dentro del periodo de retencion WAL.

RBAC (Role-Based Access Control)
  Control de Acceso Basado en Roles. Aplicado en capa de API y capa de DocType.
  Cuatro niveles: Admin Principal, Admin Secundario, Asistente, Invitado.

Secret Manager
  Servicio GCP para almacenar secretos cifrados. Inyectados como variables
  de entorno al contenedor Cloud Run en el inicio via secret_key_ref Terraform.

Sidebar
  Panel lateral permanente del Desktop Shell. Contiene funciones fijas del
  sistema (navegacion, modulos, apps nativas) y WebViews custom del usuario.

Soft Delete
  Patron de eliminacion donde los registros se marcan is_deleted=1 en vez de
  borrarse fisicamente. Aplicado universalmente en todas las entidades.

Sterling PDF
  App nativa del Desktop Shell para edicion y visualizacion profesional de PDFs.
  Invocada desde el Sidebar via IPC con cambio de vista del BrowserWindow.

Terraform
  Herramienta IaC que aprovisiona y gestiona todos los recursos GCP
  de forma determinista desde configuracion declarativa.

VPC Connector
  Conector de Acceso VPC Serverless de GCP. Permite a Cloud Run comunicarse
  con recursos en la red privada VPC, especificamente la IP privada de Cloud SQL.

VolView 3D
  App nativa del Desktop Shell para visualizacion medica 3D. Invocada desde
  el Sidebar via IPC. Reemplaza el WebView principal al montarse.

WebGPU
  API de Node.js/browser para computo GPU acelerado. Ruta de inferencia
  alternativa para Gemma 270M cuando hay GPU compatible disponible.

WebView
  Vista de navegador web embebida en el Desktop Shell. Base de las
  integraciones de apps custom gestionadas por el usuario en el Sidebar.

WORM (Write Once, Read Many)
  Caracteristica de inmutabilidad de almacenamiento. Aplicada al tabAudit Log
  y GCP Cloud Logging. Garantiza integridad del rastro de auditoria para
  cumplimiento normativo clinico.
